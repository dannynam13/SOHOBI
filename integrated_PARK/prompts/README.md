## 1. 폴더 구조

```
prompts/
├── README.md                        ← 지금 읽는 이 파일
├── signoff_admin/
│   └── evaluate/
│       ├── skprompt.txt             ← 행정 도메인 루브릭 (A1~A5)
│       └── config.json              ← 모델 설정 (temperature, max_tokens)
├── signoff_finance/
│   └── evaluate/
│       ├── skprompt.txt             ← 재무 도메인 루브릭 (F1~F5)
│       └── config.json
└── signoff_legal/
    └── evaluate/
        ├── skprompt.txt             ← 법률 도메인 루브릭 (G1~G4)
        └── config.json
```

각 폴더는 **해당 도메인 팀원이 수정 책임**을 집니다.

| 팀원 | 담당 폴더 | 평가 항목 |
|---|---|---|
| CHOI | `signoff_legal/` | G1~G4 (면책 조항, 법령 인용 등) |
| CHANG | `signoff_finance/` | F1~F5 (수치, 가정 조건, 리스크 등) |
| NAM | `signoff_admin/` | A1~A5 (법령, 서식, 절차, 기관, 기한) |
| PARK | 공통(C1~C5) 포함 전체 관리 | C1~C5 (응답성, 완결성, 일관성 등) |

## 2. 루브릭이란 무엇인가

Sign-off Agent는 하위 에이전트(AdminAgent, FinanceAgent, LegalAgent)가 생성한 **응답 초안(draft)** 을 읽고, 아래 질문에 JSON으로 답합니다.

```
"이 응답이 우리 서비스의 품질 기준을 통과하는가?"
```

기준을 통과하면 `approved: true`, 통과 못 하면 `approved: false` 와 함께 **무엇을 고쳐야 하는지 지시문(retry_prompt)** 을 반환합니다. 그러면 에이전트가 지시문을 반영해 응답을 다시 작성하고, 최대 3회까지 재시도합니다.

```
에이전트 응답 생성
       ↓
Sign-off 평가 (루브릭 검증)
       ↓
    approved?
   ┌────┴─────┐
  YES         NO
   ↓           ↓
최종 응답    retry_prompt 생성
              ↓
         에이전트 재작성
              ↓
        (최대 3회 반복)
              ↓
         escalated 반환
```

루브릭을 **너무 엄격하게** 쓰면 항상 escalated가 됩니다.
루브릭을 **너무 느슨하게** 쓰면 품질이 낮은 응답도 통과됩니다.
**딱 맞는 기준** 을 찾는 것이 루브릭 작업의 핵심입니다.

## 3. 루브릭 파일 읽는 법

`skprompt.txt` 는 Semantic Kernel의 프롬프트 파일 형식으로 작성됩니다. 구조는 다음과 같습니다.

```xml
<message role="system">
당신은 [도메인] 도메인 AI 에이전트 응답의 품질을 검증하는 Sign-off Agent입니다.

[평가 규칙 — 반드시 준수]
1. 모든 항목을 평가해야 한다.
...

[L1 공통 루브릭]
- C1 ...
- C2 ...
...

[L2 [도메인] 도메인 루브릭]
- [항목코드] ...
...

[출력 형식 — JSON만 출력]
{
  "approved": true/false,
  "passed": [...],
  "issues": [...],
  "retry_prompt": "..."
}
</message>

<message role="user">
평가할 draft:
{{$draft}}
</message>
```

`{{$draft}}` 자리에 에이전트가 생성한 응답 전문이 삽입됩니다. 루브릭 파일을 수정할 때는 `{{$draft}}` 는 절대 건드리지 마세요.

---

### config.json 구조

```json
{
  "schema": 1,
  "execution_settings": {
    "default": {
      "max_tokens": 1000,
      "temperature": 0.0,
      "response_format": { "type": "json_object" }
    }
  }
}
```

- `temperature: 0.0` — 평가의 재현성을 위해 반드시 0으로 유지합니다. 올리면 같은 응답에 다른 판정이 나옵니다.
- `max_tokens: 1000` — Sign-off 출력(JSON 판정문)에 충분한 토큰입니다. 늘려도 무방하지만 줄이면 JSON이 잘릴 수 있습니다.
- `response_format` — JSON 출력 강제. 절대 변경하지 마세요.

## 4. 항목별 평가 기준 상세 설명

### L1 공통 루브릭 (C1~C5) — 전 도메인 동일

모든 도메인 루브릭에 공통으로 포함되어 있습니다. PARK가 관리하지만, 동작 방식을 이해해야 자신의 도메인 항목을 올바르게 작성할 수 있습니다.

| 코드 | 이름 | 통과 조건 | 자주 나오는 오탈 |
|---|---|---|---|
| C1 | 질문 응답성 | 사용자 질문에 실질적으로 답하고 있으면 통과 | "내용이 부족하다"는 이유로 탈락시키는 것 → 탈락 기준 아님 |
| C2 | 완결성 | 문장이 중간에 물리적으로 끊기면 탈락 | 정보가 적다는 이유로 탈락 → 탈락 기준 아님 |
| C3 | 내부 일관성 | 응답 내 수치/사실이 서로 직접 모순되면 탈락 | 외부 사실과 다르다는 이유로 탈락 → 탈락 기준 아님 |
| C4 | 톤 적절성 | 공식적이고 신뢰할 수 있는 어조 | 주관적 판단이 개입되기 쉬움 |
| C5 | 할루시네이션 징후 | '반드시', '절대', '항상' 등 단정적 표현이 실제 존재하면 탈락 | 표현 없는데 "단정적으로 보인다"며 탈락 → 금지 |

**C2 주의**: "응답이 너무 짧다"는 이유로 C2를 탈락시키는 것은 잘못입니다. C2는 오직 **물리적 절단**(문장이 갑자기 끝남, 목록이 중간에 멈춤)에만 적용합니다.

**C3 주의**: 외부 법령·사실 확인이 안 된다는 이유로 C3를 탈락시키면 안 됩니다. 응답 텍스트 내부에서 A와 B가 서로 모순될 때만 탈락입니다.

### L2 도메인별 루브릭

#### 재무 도메인 (F1~F5) — CHANG 담당

| 코드 | 이름 | 통과 조건 | 참고 |
|---|---|---|---|
| F1 | 수치 제시 | 금액·비율 등 수치가 하나라도 있으면 통과 | 수치의 정확성은 평가 안 함 |
| F2 | 단위 표기 | 단위(원, %)가 하나라도 있으면 통과 | 모든 수치에 단위 불필요 |
| F3 | 가정 전제 언급 | 시뮬레이션 입력 조건이 하나라도 언급되면 통과 | 단일 입력값·실제 데이터 기반 모두 인정 |
| F4 | 손실 확률 명시 | 손실 발생 확률이 수치(%)로 명시되면 통과 | **핵심 항목** — 수치 없는 정성 표현만으로는 탈락 |
| F5 | 리스크 경고 | 리스크 관련 표현이 하나라도 있으면 통과 | 리스크를 **명시적으로 부정**할 때만 탈락 |

#### 법률 도메인 (G1~G4) — CHOI 담당

| 코드 | 이름 | 통과 조건 |
|---|---|---|
| G1 | 면책 조항 | "법적 조언 아님" 등 면책 표현이 하나라도 있으면 통과 |
| G2 | 법령 기준 시점 | 개정 가능성 언급이 하나라도 있으면 통과 |
| G3 | 전문가 상담 권고 | 상담 권고 표현이 하나라도 있으면 통과 |
| G4 | 법령명·조항 인용 | 법령명 또는 조항 번호가 하나라도 있으면 통과 |

#### 행정 도메인 (A1~A5) — NAM 담당

| 코드 | 이름 | 통과 조건 |
|---|---|---|
| A1 | 법령 조항 인용 | 법령명 또는 조항 번호가 하나라도 있으면 통과 |
| A2 | 서식 번호 언급 | 서식명 또는 양식명이 하나라도 있으면 통과 |
| A3 | 절차 단계 서술 | 단계 또는 순서 서술이 하나라도 있으면 통과 |
| A4 | 관할 기관 언급 | 기관명이 하나라도 있으면 통과 |
| A5 | 기한 정보 언급 | 처리 기한 등 기한 관련 표현이 하나라도 있으면 통과 |

## 5. 루브릭 수정 방법

### 새 항목 추가하기

예를 들어 법률 도메인에 "상담 기관 연락처 제공 여부"를 추가하고 싶다면:

**1단계**: `signoff_legal/evaluate/skprompt.txt` 의 평가 규칙에 항목 코드를 추가합니다.

```diff
- 아래 루브릭의 모든 항목 C1, C2, C3, C4, C5, G1, G2, G3, G4를 반드시 평가해야 한다.
+ 아래 루브릭의 모든 항목 C1, C2, C3, C4, C5, G1, G2, G3, G4, G5를 반드시 평가해야 한다.
```

**2단계**: `[L2 법률 도메인 루브릭]` 아래에 항목 설명을 추가합니다.

```
- G5 상담 기관 연락처 존재 여부: 응답에 구체적인 법률 상담 기관 연락처(전화번호, 웹사이트 등)가
  하나라도 있으면 통과. 연락처가 전혀 없는 경우에만 issues로 분류한다.
```

**3단계**: `signoff_agent.py` 의 `REQUIRED_CODES` 딕셔너리에 새 코드를 추가합니다.

```python
# signoff/signoff_agent.py
REQUIRED_CODES = {
    "admin":   {"C1", "C2", "C3", "C4", "C5", "A1", "A2", "A3", "A4", "A5"},
    "finance": {"C1", "C2", "C3", "C4", "C5", "F1", "F2", "F3", "F4", "F5"},
    "legal":   {"C1", "C2", "C3", "C4", "C5", "G1", "G2", "G3", "G4", "G5"},  # G5 추가
}
```

> 이 세 곳이 항상 일치해야 합니다. 하나라도 빠지면 Sign-off가 오작동합니다.

---

### 기존 항목 기준 강화하기

"가정 전제를 하나만 언급하는 것으로는 부족하고, 반드시 세율을 포함해야 한다"고 강화하려면:

```diff
- F3 가정 전제 언급 존재 여부: 응답에 계산 또는 시뮬레이션의 가정이 하나라도 언급되어 있으면 통과.
+ F3 가정 전제 언급 존재 여부: 응답에 세율을 포함한 시뮬레이션 가정이 언급되어 있으면 통과.
+   세율 언급이 없는 경우에만 issues로 분류한다.
```

**주의**: 기준을 강화할수록 에이전트 응답 프롬프트(`agents/finance_agent.py` 의 `_EXPLAIN_PROMPT`)에도 해당 요건을 지시해야 합니다. **루브릭과 에이전트 프롬프트는 항상 쌍으로 수정**합니다.

---

### 기존 항목 완화하기

루브릭이 너무 엄격해서 정상 응답이 계속 탈락할 때, 기준을 완화합니다.

```diff
- F5 리스크 경고 언급 존재 여부: ... 리스크를 부정하거나 축소하는 표현은 인정하지 않는다.
+ F5 리스크 경고 언급 존재 여부: ... 리스크를 부정하거나 축소하는 표현은 인정하지 않는다.
+ 중요: 시뮬레이션 수치(예: '손실 확률 0%')가 포함되어 있더라도, 그 외에 리스크 관련 표현이
+ 하나라도 있으면 F5 통과로 분류한다. 시뮬레이션 결과 수치 자체는 리스크 부정으로 간주하지 않는다.
```

## 6. 실제 개선 사례 — 실패에서 배운 것들

이 시스템을 처음 테스트했을 때 재무 에이전트 응답이 **항상 escalated** 였습니다. 로그를 통해 원인을 추적하고 3단계에 걸쳐 수정했습니다. 각 사례를 그대로 기록합니다.

---

### 사례 1: F3·F4 탈락 — 루브릭과 에이전트 프롬프트의 불일치

#### 문제

재무 에이전트 응답이 매 시도마다 F3("가정 전제 없음"), F4("불확실성 없음")로 탈락했습니다.

```
attempt 1: issues=['F3', 'F4']
  F3: 시뮬레이션에 사용된 가정 조건이 응답에 명시되지 않았습니다.
  F4: 예측의 불확실성이나 신뢰 구간 관련 표현이 없습니다.
```

#### 원인

`finance_agent.py` 의 `_EXPLAIN_PROMPT` 가 LLM에게 시뮬레이션 입력값(매출·원가·급여)을 넘기지 않았습니다.

```python
# 수정 전: 결과만 넘김
_EXPLAIN_PROMPT = """
- 평균 월 순이익: {avg_profit:,}원
- 손실 발생 확률: {loss_prob:.1%}
"""
```

LLM은 "어떤 가정으로 계산했는지" 모르니까 응답에 가정을 쓸 수 없었습니다. 루브릭은 "가정을 써라"고 요구하는데, 에이전트는 가정 데이터 자체를 모르는 상태였습니다.

#### 수정 — 에이전트 프롬프트와 시뮬레이션 통계량 확장

**`plugins/finance_simulation_plugin.py`**:
```python
# 수정 전: 2개 반환
return {"average_net_profit": round(avg), "loss_probability": round(loss_prob, 4)}

# 수정 후: 5개 반환 (표준편차 + 신뢰구간 추가)
return {
    "average_net_profit": round(avg),
    "loss_probability":   round(loss_prob, 4),
    "std_profit":         round(std),       # 표준편차
    "p5_net_profit":      round(p5),        # 하위 5%
    "p95_net_profit":     round(p95),       # 상위 95%
}
```

**`agents/finance_agent.py`**:
```python
# 수정 후: 가정 조건 문자열 구성 후 프롬프트에 전달
assumption_lines = [
    f"- 월매출: {rev_str} (±10% 정규분포 가정)",
    f"- 원가: {variables.get('cost', 0):,}원 (±10% 정규분포 가정)",
    f"- 급여: {variables.get('salary', 0):,}원",
    f"- 세율: {variables.get('tax_rate', 0.2):.0%}",
]
# _EXPLAIN_PROMPT 에 {assumptions}, {std_profit}, {p5}, {p95} 추가
```

#### 결과

F3, F4 모두 통과. 에이전트 응답에 "월매출 7,000,000원, ±10% 정규분포 가정", "90% 신뢰구간 1,012,884원~2,965,805원" 이 명시됩니다.

**교훈**: 루브릭이 요구하는 것이 에이전트 프롬프트에 데이터로 제공되지 않으면, 아무리 retry해도 해결되지 않습니다. **루브릭의 각 항목은 에이전트 프롬프트에 대응하는 지시가 있어야** 합니다.

### 사례 2: C5·F5 반복 탈락 — 루브릭이 통계 수치를 단정적 표현으로 오판

#### 문제

F3·F4를 고쳤는데도 C5, F5로 계속 탈락했습니다.

```
attempt 1: issues=['C5', 'F5']
  C5: '손실 발생 확률이 0%'라는 표현이 단정적입니다.
  F5: 리스크 경고가 부족합니다. '손실 발생 확률 0%'라는 표현으로 리스크가 축소됩니다.
```

#### 원인

입력값(매출 700만원, 원가 200만원, 급여 250만원)으로 시뮬레이션하면 **수학적으로 손실확률 0%가 정확한 결과**입니다. 하지만 루브릭의 C5와 F5는 "0%"라는 수치를 할루시네이션 또는 리스크 부정으로 판정했습니다.

이것은 루브릭의 설계 문제입니다. 시뮬레이션 통계 결과와 에이전트의 "보장" 발언을 구분하지 못하고 있었습니다.

#### 수정 — 루브릭에 예외 조항 추가

**`signoff_finance/evaluate/skprompt.txt`**:

```diff
  C5 할루시네이션 징후 부재: 응답에 단정적 표현이 실제로 존재하는 경우에만 위반.
+ 중요: 몬테카를로 시뮬레이션의 통계적 결과로 제시된 수치(예: '시뮬레이션 결과 손실 발생 확률 0%')는
+ 단정적 표현이 아니다. 해당 수치가 시뮬레이션 가정 범위 내의 결과임이 맥락상 명확하면 C5 passed로 분류한다.

  F5 리스크 경고 언급 존재 여부: 리스크 관련 표현이 하나라도 있으면 통과.
  단, '원금 손실 없이', '반드시 안전합니다' 등 리스크를 명시적으로 부정하는 표현만 인정하지 않는다.
+ 중요: 시뮬레이션 수치(예: '손실 확률 0%')가 포함되어 있더라도, 그 외에 리스크 관련 표현이 하나라도 있으면
+ F5 통과로 분류한다. 시뮬레이션 결과 수치 자체는 리스크 부정으로 간주하지 않는다.
```

#### 결과

동일 응답이 C5, F5 모두 통과. retry_count 1~2회로 감소.

**교훈**: 루브릭 작성 시, **도메인 고유의 정상 패턴**(예: 시뮬레이션 결과의 0%)이 오탈될 수 있습니다. 실제 테스트 전에는 이런 예외가 잘 보이지 않습니다. **실제 에이전트 응답을 보고 루브릭을 조정**해야 합니다.

### 사례 3: C2 탈락 — 응답이 중간에 잘림

#### 문제

간헐적으로 C2("응답 텍스트가 물리적으로 절단되었습니다")로 탈락했습니다.

```
attempt 2: issues=['C2']
  C2: 응답 텍스트가 '재 작업...을 유연하게 조정하는 것이 중요합니다'에서 비정상적으로 끝났으며,
      문장이 완결되지 않은 채 종료됨
```

#### 원인

`finance_agent.py` 의 `_call_llm` 에 `max_tokens` 가 설정되어 있지 않았습니다. 재무 설명은 길게 작성되는데, 기본 max_tokens 한도에서 응답이 잘렸습니다.

```python
# 수정 전
settings = OpenAIChatPromptExecutionSettings(temperature=0.3)

# 수정 후
settings = OpenAIChatPromptExecutionSettings(temperature=0.3, max_tokens=3000)
```

#### 교훈

### 사례 4: I_NEED_DRAFT — _EXPLAIN_PROMPT 템플릿 구조가 빈 응답 유발

#### 문제

모델 교체 후 재무 쿼리가 재시도 3회, 거부 4회로 escalated 되었습니다. 로그를 보니 시도 1·3에서 모든 항목이 "Draft not provided" 로 탈락하고, `final_draft` 에는 투자 회수 구간 텍스트만 남아 있었습니다.

```
attempt 1: issues=['C1','C2','C3','C4','C5','F1','F2','F3','F4','F5']
  C1: Draft not provided; cannot evaluate
attempt 3: issues=['I_NEED_DRAFT_C1', ..., 'I_NEED_DRAFT_F5']
  I_NEED_DRAFT_C1: 평가 대상 draft가 제공되지 않아 C1을 평가할 수 없습니다.

final_draft: "\n\n[투자 회수 전망]\n투자 회수 가능 여부는 True이며, 회수까지 약 19개월..."
```

#### 원인

`_EXPLAIN_PROMPT` 끝에 다음 템플릿 구조가 있었습니다.

```
응답 형식:
[사용자 질문]
{question}

[에이전트 응답]
(위 기준을 충족하는 응답 내용)
```

새 모델이 이 구조를 "채워야 할 폼"으로 해석해 `[에이전트 응답]` 헤더만 출력하고 내용 없이 종료했습니다. `str(result)` 가 빈 문자열을 반환하고, 4단계(투자 회수) 텍스트만 draft에 붙어 sign-off로 넘어갔습니다.

재시도 때 sign-off의 retry_prompt("draft 본문을 붙여 주세요")가 `_RETRY_PREFIX` 로 앞에 붙으면 모델이 실제 내용을 생성했고 (시도 2·4), F3·F5 수정 지시가 붙으면 다시 빈 응답을 반환하는 교대 패턴이 발생했습니다.

#### 수정 — 템플릿 구조 제거

```diff
- 응답 형식:
- [사용자 질문]
- {question}
-
- [에이전트 응답]
- (위 기준을 충족하는 응답 내용)
+ 위 분석 결과에 기반하여 다음 질문에 답변하십시오:
+
+ 질문: {question}
+
+ 작성 기준:
+ - 첫 단락에 위의 가정 조건(월매출, 원가, 급여 등)을 명시하십시오.
+ ...
```

#### 교훈 — 응답 형식 지시 방식

**`[헤더]\n(placeholder)` 형식의 응답 형식 지시는 모델이 템플릿 완성 태스크로 오해할 수 있습니다.** 응답 형식을 지시할 때는 "이렇게 출력하라"는 명령형 문장을 사용하고 빈 칸 형식은 피하십시오.

### 사례 5: F3·F5 반복 탈락 — 루브릭이 시뮬레이션 가정을 금융 가정으로 오판

#### 문제

사례 4의 빈 응답 문제를 제외하고, 실제 draft가 생성된 시도(2·4)에서도 F3·F5가 반복 탈락했습니다.

```
attempt 2: issues=['F3', 'F4', 'F5']
  F3: 가정 전제 언급이 부재. 가정(금리, 물가상승률, 할인율 등) 명시 필요.
  F5: 리스크 경고 부재: 원금 손실 가능성, 시장 리스크 등에 대한 경고가 없음.
```

draft 에는 `±10% 정규분포 가정`, `경기 침체·임대료 급등·수요 급감 등의 외부 충격` 등이 명시되어 있었습니다.

#### 원인

F3 루브릭의 예시 `(금리 고정, 물가상승률 등)` 가 sign-off LLM을 **투자금융 가정**에만 고정시켰습니다. 몬테카를로 가정(`±10% 정규분포`)은 유효한 가정임에도 탈락 처리했습니다.

F5 루브릭은 유효한 리스크 표현 예시가 없어 `경기 침체`, `외부 충격` 같은 서술을 리스크 경고로 인식하지 못했습니다.

#### 수정 — 루브릭에 시뮬레이션 가정 인정 조항 및 유효 표현 예시 추가

```diff
  F3 가정 전제 언급 존재 여부: 응답에 계산 또는 시뮬레이션의 가정이 하나라도 언급되면 통과.
+ 중요: 몬테카를로 시뮬레이션의 분산 가정(예: '매출·원가 ±10% 정규분포 가정')은 유효한 가정으로 인정.
+ '±10%', '정규분포 가정', '시뮬레이션 가정 범위' 등이 있으면 F3 통과.
+ 금리·할인율·물가상승률이 없어도 시뮬레이션 가정이 있으면 통과.

  F5 리스크 경고 언급 존재 여부: 리스크 관련 표현이 하나라도 있으면 통과.
+ 유효한 리스크 표현 예시: '경기 침체', '임대료 급등', '수요 급감', '외부 충격',
+   '손실로 이어질 수 있', '시장 변동성', '원금 손실 가능성', '실제 손실', '비용 급증' 등.
```

#### 교훈

#### 교훈 — 루브릭 예시 표현의 암묵적 제약

루브릭의 **예시 표현은 허용 범위를 암묵적으로 제한**합니다. 도메인 고유 표현(시뮬레이션 가정, 통계적 리스크 서술)이 있다면 루브릭에 명시적으로 예시를 추가해야 합니다. 실제 에이전트 응답을 로그에서 확인하고 루브릭을 맞춰 나가십시오.

### 사례 6: F4 기준 재정의 — "신뢰구간"에서 "손실 확률 수치 명시"로 교체

#### 배경

재무 시뮬레이션이 사용자 단일 입력 대신 **DB에서 가져온 실제 매장 매출 데이터** 를 사용하도록 발전하면서, 다음 문제가 제기되었습니다.

- 실제 매장 데이터의 분포는 정규분포를 따르지 않으므로, 응답에서 `±10% 정규분포 가정` 등의 표현은 의미가 없다.
- 신뢰구간·표준편차보다 **손실 발생 확률(n%)** 이 창업자에게 훨씬 중요한 지표다.
- F4가 "신뢰구간 표현 존재 여부"를 기준으로 삼으면, 실제로 중요한 손실 확률을 빠뜨려도 통과된다.

#### 수정 내용

**`signoff_finance/evaluate/skprompt.txt`**:

```diff
- F4 불확실성 언급 존재 여부: 응답에 예측·시뮬레이션 결과의 불확실성 또는
-   신뢰 구간 관련 표현이 하나라도 있으면 통과.
+ F4 손실 확률 명시 여부: 응답에 시뮬레이션에서 산출된 손실 발생 확률이
+   수치(%)로 명시되어 있으면 통과.
+   수치 없이 정성적 표현('낮습니다', '안전합니다')만 있는 경우 issues로 분류한다.
+   단, 시뮬레이션 범위 내 손실이 관측되지 않았음을 서술형으로 명시한 경우도 통과.

  F3 가정 전제 언급 존재 여부: ...
- 중요: '±10%', '정규분포 가정', '시뮬레이션 가정 범위' 등의 표현이 있으면 F3 통과.
+ 중요: 단일 입력값('월매출', '원가', '임대료' 등 수치 언급) 또는
+   실제 데이터 기반('실제 매장', 'n개 데이터', 'DB 데이터' 등) 표현이 있으면 통과.

  핵심 항목:
- 핵심 항목(C1·C3·F1·F3)이 기준 미달이면 issues로 분류한다.
+ 핵심 항목(C1·C3·F1·F3·F4)이 기준 미달이면 issues로 분류한다.
```

**`agents/finance_agent.py`**:

```diff

- - 평균 월 순이익: {avg_profit:,}원
- - 표준편차: {std_profit:,}원
```

#### 교훈 — 루브릭 항목과 실제 비즈니스 우선순위의 정렬

루브릭 항목은 응답 품질을 평가하는 기준이지만, **창업자에게 실제로 중요한 정보가 무엇인지** 를 반영해야 합니다. 도메인 전문가(재무 담당 팀원)의 피드백을 받아 루브릭의 초점을 재정렬하는 것이 이번 개선의 핵심입니다.

## 7. 테스트 실행 방법

### 방법 A: 전체 파이프라인 테스트 (서버 사용)

서버를 실행한 후 curl 로 테스트합니다.

```bash
# 서버 실행 (integrated_PARK/ 폴더에서)
.venv/bin/python3 -m uvicorn api_server:app --port 8000 --reload

# 재무 도메인 테스트
curl -X POST http://localhost:8000/api/v1/query \

  -d '{"question": "월 매출 700만원, 원가 200만원, 급여 250만원 수익 시뮬레이션"}'

# 응답에서 확인할 항목
# - "status": "approved" 이면 루브릭 통과
# - "retry_count": 0이면 첫 시도에 통과, 숫자가 클수록 수정이 많이 필요했음
# - "status": "escalated" 이면 3회 초과 탈락 → 로그 확인 필요
```

### 방법 B: Sign-off 단독 테스트 (루브릭만 테스트할 때)

이미 작성한 draft 텍스트를 Sign-off에만 통과시켜 어느 항목에서 탈락하는지 확인합니다.

```bash
# 예시: 법률 응답 Sign-off 단독 검증
curl -X POST http://localhost:8000/api/v1/signoff \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "legal",
    "draft": "[에이전트 응답]\n본 응답은 법적 조언이 아닙니다. 음식점 창업 시 임대차보호법 제3조에 따라..."
  }'

# 응답 예시
# {
#   "approved": false,
#   "passed": ["C1", "C2", "C3", "C4", "C5", "G1", "G2", "G4"],
#   "issues": [{"code": "G3", "reason": "전문가 상담 권고 표현이 없습니다."}],
#   "retry_prompt": "법률 전문가 또는 법률구조공단 상담을 권고하는 문구를 추가하세요."
# }
```

이 방법은 **루브릭만 집중적으로 디버그할 때** 유용합니다. 전체 에이전트를 거치지 않아서 빠릅니다.

### 방법 C: 로그로 탈락 이력 확인

서버를 통해 테스트한 경우, `logs/` 폴더에 자동으로 로그가 남습니다.

```bash
# 최근 요청의 탈락 이력 확인
python3 -c "
import json

with open('logs/rejections.jsonl') as f:
    for line in f:
        r = json.loads(line)
        print(f'[{r[\"ts\"][11:19]}] {r[\"domain\"]} retry={r[\"retry_count\"]}')
        for rej in r['rejection_history']:
            codes = [i['code'] for i in rej['issues']]
            print(f'  attempt {rej[\"attempt\"]}: {codes}')
            for issue in rej['issues']:
                print(f'    {issue[\"code\"]}: {issue[\"reason\"][:80]}')
"
```

로그 파일 위치:
- `logs/queries.jsonl` — 모든 요청 (통과 포함)
- `logs/rejections.jsonl` — 탈락 이력이 있는 요청만

## 8. 자주 하는 실수와 해결법

### 실수 1: 루브릭만 수정하고 `REQUIRED_CODES` 를 업데이트 안 함

`signoff_agent.py` 의 `REQUIRED_CODES` 에 새 코드가 없으면, Sign-off가 "이 항목이 누락됐다"며 재평가 루프에 빠집니다.

```python
# 반드시 같이 수정
REQUIRED_CODES = {
    "legal": {"C1", "C2", "C3", "C4", "C5", "G1", "G2", "G3", "G4", "G5"},  # G5 추가
}
```

---

### 실수 2: 루브릭을 강화했는데 에이전트 프롬프트를 안 고침

루브릭에 새 요건을 추가했지만 에이전트가 해당 내용을 응답에 쓸 이유가 없으면, 항상 탈락합니다. **루브릭 변경 = 에이전트 프롬프트도 함께 검토**해야 합니다.

담당 에이전트 파일:
- 행정 → `agents/admin_agent.py` 의 `SYSTEM_PROMPT`
- 재무 → `agents/finance_agent.py` 의 `_EXPLAIN_PROMPT`
- 법률 → `agents/legal_agent.py` 의 `SYSTEM_PROMPT`

---

### 실수 3: "판단이 애매하면 issues로 분류하라"는 규칙을 너무 넓게 해석

루브릭에는 "판단이 애매한 경우 issues로 분류하라"는 규칙이 있지만, 이것은 **항목 자체의 충족 여부가 불명확할 때**를 의미합니다. "더 잘 쓸 수 있겠다" 싶을 때 issues로 넣으면 안 됩니다.

각 항목 설명의 "~인 경우에만 issues로 분류한다"는 문구가 실제 탈락 조건입니다. 이 조건에 해당하지 않으면 passed입니다.

---

### 실수 4: temperature를 0 이상으로 올림

Sign-off `config.json` 의 temperature를 올리면, 동일한 응답을 두 번 평가했을 때 다른 결과가 나올 수 있습니다. 루브릭 테스트의 재현성을 위해 반드시 `0.0` 을 유지합니다.

---

### 실수 5: `{{$draft}}` 를 수정함

`skprompt.txt` 에서 `{{$draft}}` 는 에이전트 응답이 삽입되는 자리 표시자입니다. 이 부분을 지우거나 바꾸면 Sign-off가 아무 응답도 평가하지 못합니다. 절대 수정하지 마세요.
